# Risks And Assumptions

## Must Confirm Before First Implementation

- 选定的 OpenClaw 宿主必须能加载 skill bundle，并向 skill 暴露最低必需能力：浏览器导航、页面读取和文件输出。
- 操作者必须能够以合法、可复用的方式访问第一版要验证的 1688 搜索页、产品页和供应商页；如果需要登录，这个会话由操作者自己提供。

## Assumptions

- 第一版允许采用低频、浏览器优先的研究方式，而不是要求高吞吐量批量采集。
- 第一版的合规边界已经固定：只做浏览器可见页面上的低频研究，不做验证码绕过、平台限制规避、隐藏接口逆向或批量化站点采集。
- 第一版默认输出是固定结构的 Markdown 研究报告；JSON、CSV 和数据库落盘都降级到后续增强。
- 第一版以供应商筛选为主要目标，代表产品只作为证据和上下文，不单独定义成产品情报项目。
- 操作者对 1688 的访问方式、账号状态和使用边界由人工负责，第一版 skill 不接管账号或会话恢复。

## Risks

- Risk: 1688 页面结构、登录要求、验证码或地区限制可能让 skill 无法稳定读取目标信息。
  Mitigation: 第一版显式支持降级和失败说明；优先浏览器优先流程，不把隐藏接口或高自动化假设写死在 MVP 中。

- Risk: “优质供应商”本身不是页面直接给出的真值，很多判断只能基于有限的可见信号。
  Mitigation: 建立保守 rubric，把页面事实、启发式判断和未知项分开写，避免伪精确打分。

- Risk: 宿主应用的实际工具集合可能和预期不同，例如没有稳定浏览器能力，或 skill 路径约定不同。
  Mitigation: 在真正开始实现前先验证最小宿主能力和目录约定；如果不满足，就报告 blocker，而不是扩展成平台改造项目。

- Risk: 平台规则、合规边界或团队内部使用规范可能不允许“抓取”语义过重的实现。
  Mitigation: MVP 已固定为供应商研究助手；只做低频、可审计、操作者在环的研究流，并把不合规方案明确排除在范围外。

## Open Questions

- 本次 MVP 将落在哪个具体 OpenClaw 宿主环境里，这个环境是否真的向 skill 暴露了浏览器导航、页面读取和文件输出这组最小能力？
- 在第一版验证时，操作者是否能提供一个合法、可复用的 1688 访问路径，使搜索页、产品页和供应商页都可实际打开和读取？
